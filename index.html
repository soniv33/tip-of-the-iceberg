🧊 tip-of-the-iceberg

A hands-on tour of how an Apache Iceberg table actually works. Query it, write to it, delete, update, compact, time-travel — and watch the metadata tree and files react.

Explain it like I'm five
  1. Movies live in boxes (files). Each box's lid says the first & last title inside (min/max), and which genre shelf (partition) it's on.
  2. To find something, we read lids first and skip boxes that can't contain it.
  3. Every change writes a new snapshot (a photo of the shelves). Old photos never change, so we can time-travel.
  4. Deleting either rewrites a box (copy-on-write) or drops a "skip this row" sticky note (merge-on-read).
  5. Compaction tidies many small boxes into big ones; expiry throws away old photos and unused boxes.

➕ Append a row (an Iceberg insert)

Writes a new data file in the row's genre partition and a new snapshot. Old files/manifests are reused — appends are cheap.

🗑️ Delete rows (copy-on-write vs merge-on-read)

Delete where a column equals a value. Copy-on-write rewrites the data files without those rows. Merge-on-read writes a tiny delete file and hides the rows at read time (folded away later by compaction).

✏️ Update / Upsert

Update sets a column for matching rows (copy-on-write delete+insert). Upsert merges a row by title: existing title is replaced, a new title is inserted (MERGE).

🧹 Maintenance — compact, expire, evolve

Compact merges each partition's small files into one and folds in delete files. Expire keeps only the latest snapshot and garbage-collects unreferenced files. Add column evolves the schema with no data rewrite (old rows read null).
